# portal target runtime

Reference and demo project for `MCU UI Studio for LVGL`.

German version: [README.de.md](README.de.md)

This target project shows one practical way to integrate editor-generated LVGL UI
code into an ESP-IDF project with FreeRTOS, message queues, and clearly
separated runtime modules.

Important:

- this target project is a template, not a mandatory architecture
- the editor should support multiple integration styles over time
- this repository intentionally demonstrates the RTOS/message-based path

The goal is to show how generated UI code can be integrated into an embedded
runtime without pushing application logic into raw LVGL callbacks.

## What This Project Is For

This project is intended as:

- a demo for `MCU UI Studio for LVGL`
- a reference for RTOS-based integration
- a starting point for custom templates
- an example of how generated code and handwritten runtime code can stay cleanly separated

This project is not intended as:

- the only valid architecture for all targets
- a recommendation for systems without RTOS
- a complete production reference for every embedded application

Other integration styles are absolutely possible, for example:

- direct logic inside event callbacks
- a simple main loop without RTOS
- project-specific HAL or fieldbus integration

## Core Idea

The editor generates UI code.

The runtime decides how that code is integrated into the actual embedded
application.

In this template that means:

- generated UI files live under `main/generated/`
- handwritten runtime files live under `main/`
- module communication happens through FreeRTOS queues
- application logic stays outside generated LVGL code

## Architecture In This Template

The data flow intentionally follows a closed RTOS pattern:

```text
Display (LVGL)
  -> control_queue
Controller
  -> fieldbus_queue
Fieldbus / Hardware
  -> control_queue
Controller
  -> display_queue
Display (LVGL)
```

The roles are clearly separated:

- `display`
  wraps access to the generated UI update functions
- `controller`
  makes application decisions and mediates between display and hardware
- `fieldbus`
  executes hardware actions and reports state back
- `machine`
  contains the static mapping between UI contract objects and hardware objects
- `message`
  defines the queue messages

## Why This Architecture

This template prefers an RTOS/message architecture because it is easy to follow
for larger MCU applications:

- events are processed in a decoupled way
- display logic and hardware logic stay separated
- controller code does not need direct LVGL calls
- the structure scales better than pure callback logic

That does not mean every application should be built like this.

## Project Layout

```text
main/
├── generated/        # UI files generated by the editor
├── include/          # Headers of the handwritten runtime
├── controller.c      # Application logic / mediation
├── display.c         # Display runtime, calls generated update API
├── display_font.c    # Runtime font for umlauts
├── fieldbus.c        # Hardware / GPIO layer
├── machine.c         # Binding between UI contract and hardware objects
├── message.c         # Queue creation
├── main.c            # BSP start, display start, task creation
└── Arial.ttf         # Embedded TTF font for umlauts

docs/
├── architecture.md
└── ui-fieldbus-integration.md
```

## Generated Code vs Runtime Code

The central separation in this project is:

- the editor generates structure, layout, event bindings, and update targets
- the runtime decides what those events mean in application terms

Typical generated files:

- `ui_start.c`
- `ui_start.h`
- `ui_start_contract.h`
- `ui_start_event.c`
- `ui_start_update.c`

Typical handwritten files:

- `controller.c`
- `display.c`
- `fieldbus.c`
- `machine.c`
- `message.c`

## What The MCU Developer Actually Changes

After regenerating the UI, these are usually the relevant places:

1. `main/machine.c`
   maps the current UI contract to real hardware objects

2. `main/fieldbus.c`
   decides what a UI action does on the hardware side

3. `main/controller.c`
   decides which feedback is sent back to the display

The MCU developer does not need to edit the generated LVGL layout code by hand,
but instead works through the contract and the runtime modules.

## Typical Runtime Flow

Example:

1. A generated button triggers an LVGL event.
2. `ui_start_event.c` builds a typed contract message and sends it to `control_queue`.
3. `controller.c` validates or routes the request.
4. `fieldbus.c` applies the hardware action.
5. A status message returns through the controller.
6. `display.c` forwards the update to generated `ui_start_update_*` functions.

This keeps raw LVGL details out of the controller and fieldbus logic.

## Build

This project is an ESP-IDF CMake project for `esp32p4`.

Important points:

- target: `esp32p4`
- framework: ESP-IDF
- UI: LVGL 9.4
- VS Code reference setup: `.vscode/settings.json`

## Current Scope Of `RTOS-Messages`

The currently tested and reliable scope of this template is:

- one project
- one active generated screen
- one matching contract, for example `ui_start_contract.h`

Multiple screens are conceptually prepared because the generator can emit files
such as `ui_screen2_contract.h`.

For this version, however, the intended statement is:

- the single-screen case is the recommended and tested path
- multi-screen projects are not yet systematically validated
- possible collisions or integration issues between multiple screen contracts
  still need dedicated testing

To build with the local helper script:

```bash
zsh ./run.sh build
zsh ./run.sh flash
zsh ./run.sh monitor
```

`run.sh` loads the local ESP-IDF environment and then calls `idf.py`.

## Fonts / Umlauts

In this setup the built-in LVGL Montserrat font does not contain all required
German glyphs by default.

Because of that, `main/Arial.ttf` is embedded and loaded at display startup
through `display_font.c`.

## Note About `LV_ATTRIBUTE_FAST_MEM`

`CONFIG_LV_ATTRIBUTE_FAST_MEM_USE_IRAM` remains enabled.

For this project the LVGL definition of `LV_ATTRIBUTE_FAST_MEM` was adjusted
locally so that conflicting `.iram1.x` sections from `IRAM_ATTR` are avoided.

That local adjustment is located in:

- `managed_components/lvgl__lvgl/env_support/cmake/esp.cmake`

## Outlook

A richer domain model is intentionally not the main goal of version 1 of this
template.

The current focus is:

- the editor generates stable UI files
- the template shows a readable RTOS integration
- the user can later decide whether to integrate more directly, more simply, or
  in a more domain-driven way

Further topics such as separate domain metadata or domain enums can come later
if real user demand emerges.
